# Friends Tab — Implementation Plan

A new top-level Friends area. Users can add friends, see each other's streak / XP / ELO / rank, peek at recent activity, and invite people who don't have an account yet.

This document scopes the work, anchors it to the existing schema and codebase patterns, and flags the few sharp edges (RLS for streak visibility, invite-flow auth, real-time notifications) up-front so they're not surprises mid-build.

---

## Goals

- **Add / remove friends** in a one-tap flow (search → request → accept)
- **Friend list** showing each friend's display name, current streak, total XP, ELO + rank, and last-active time
- **Friend profile mini-page** with deeper stats (XP history sparkline, recent recordings count, current Pathway tier, last drill date)
- **Invite link** ("share this URL — they sign up and you're auto-friends")
- **Live notification chip** when a new request lands (reuse the existing `DuelRequestNotification` pattern from `SiteHeader`)
- **Pathway/Arena hooks** so friends can challenge each other (defer the actual challenge UX — link to existing Arena)

## Non-goals (cut from MVP)

- Group chat / DMs (separate feature)
- Activity feed of every drill a friend does (privacy-heavy; needs another design pass)
- Blocking / reporting (add when there's actual scale)
- Public-friend-list (Twitter-style follower graph) — friendship is symmetric & private
- Push notifications outside the app

## Data model (Supabase)

One migration. Two new tables. One RLS escape hatch.

### `friendships`

The canonical relationship row. **Symmetric** — accepted friendships have two rows (A→B accepted, B→A accepted), or one row interpreted symmetrically depending on whether we want simpler indexing or simpler writes. **Choosing one row** (lower-id user_a, higher-id user_b) keeps storage tight and queries unambiguous.

```sql
CREATE TABLE public.friendships (
  user_a UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  user_b UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  status TEXT NOT NULL CHECK (status IN ('pending', 'accepted', 'blocked')),
  requested_by UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  accepted_at TIMESTAMPTZ,
  PRIMARY KEY (user_a, user_b),
  CHECK (user_a < user_b)  -- canonical ordering so (A,B) and (B,A) can't both exist
);

CREATE INDEX idx_friendships_user_a ON public.friendships(user_a) WHERE status = 'accepted';
CREATE INDEX idx_friendships_user_b ON public.friendships(user_b) WHERE status = 'accepted';
CREATE INDEX idx_friendships_pending ON public.friendships(user_b, requested_by) WHERE status = 'pending';
```

RLS:

```sql
-- See a row if you're one of the two parties
CREATE POLICY "friendships_select_own" ON public.friendships
  FOR SELECT USING (auth.uid() = user_a OR auth.uid() = user_b);

-- Insert: must be one of the parties AND must be the requester
CREATE POLICY "friendships_insert_self" ON public.friendships
  FOR INSERT WITH CHECK (
    auth.uid() = requested_by
    AND (auth.uid() = user_a OR auth.uid() = user_b)
  );

-- Update: the OTHER party can flip pending -> accepted; either party can flip accepted -> deleted
CREATE POLICY "friendships_update_accept" ON public.friendships
  FOR UPDATE USING (auth.uid() = user_a OR auth.uid() = user_b)
  WITH CHECK (auth.uid() = user_a OR auth.uid() = user_b);

-- Delete: either party can remove the friendship (or withdraw a pending request)
CREATE POLICY "friendships_delete_either" ON public.friendships
  FOR DELETE USING (auth.uid() = user_a OR auth.uid() = user_b);
```

### `friend_invites`

For the "share this URL" flow — generated by an existing user, claimed by a brand-new sign-up.

```sql
CREATE TABLE public.friend_invites (
  token TEXT PRIMARY KEY,                                -- 16-byte url-safe random
  inviter_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  claimed_by UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  claimed_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ NOT NULL DEFAULT (now() + INTERVAL '14 days')
);

CREATE INDEX idx_friend_invites_inviter ON public.friend_invites(inviter_id);
```

RLS:

```sql
-- Inviters see their own invites; brand-new signups need to SELECT to validate
-- the token before redeeming → split read policy.
CREATE POLICY "friend_invites_select_own" ON public.friend_invites
  FOR SELECT USING (auth.uid() = inviter_id OR auth.uid() = claimed_by);

CREATE POLICY "friend_invites_insert_self" ON public.friend_invites
  FOR INSERT WITH CHECK (auth.uid() = inviter_id);

CREATE POLICY "friend_invites_delete_own" ON public.friend_invites
  FOR DELETE USING (auth.uid() = inviter_id);
```

Validation by an unauthenticated visitor (they haven't signed up yet) goes through an edge function (see §Invite flow below) — the policy doesn't allow anon reads.

### RLS change to `streaks` ⚠

This is the **sharpest edge** in the plan. The existing policy is:

```sql
CREATE POLICY "st_select_own" ON public.streaks FOR SELECT USING (auth.uid() = user_id);
```

Only the owner can read their streak. To show a friend's streak we need ONE of:

**Option A — extend the policy to allow friend reads (recommended):**

```sql
DROP POLICY "st_select_own" ON public.streaks;
CREATE POLICY "st_select_self_or_friend" ON public.streaks FOR SELECT USING (
  auth.uid() = user_id
  OR EXISTS (
    SELECT 1 FROM public.friendships f
    WHERE f.status = 'accepted'
      AND (
        (f.user_a = auth.uid() AND f.user_b = user_id) OR
        (f.user_b = auth.uid() AND f.user_a = user_id)
      )
  )
);
```

Pros: simple Supabase client query (`from("streaks").select().in("user_id", friendIds)`).
Cons: every streak read does a friendships lookup.

**Option B — `get_friend_stats(friend_ids uuid[])` RPC**

A `SECURITY DEFINER` function that internally checks friendship, then joins streaks + xp + profiles in one query. Faster (one round-trip), but more code to maintain.

Recommend **Option A** for MVP — friend lists are small (dozens max), the index on `friendships` keeps the EXISTS subquery cheap, and the simpler client code is worth it.

### Profile fields already public

`profiles` already has `"Users can view all profiles" USING (true)` — so `display_name`, `xp`, `elo`, `strengths`, `weaknesses` are readable for anyone. Good for the friend list. **Don't tighten this** in the friends migration — other features depend on it (leaderboard, arena profile popovers).

### `user_xp` — confirm public read

The migration at `20240101000000_add_user_xp_table.sql` likely already has a public SELECT policy ("Users can view all XP data"). Confirm before relying on it; if not, extend to friend-readable like the streak.

### `recordings` — keep owner-only

Don't change. Friends shouldn't be able to play each other's recordings. If we want a "friend has practised today" indicator, derive it from `streaks.last_day` (already friend-readable after Option A) or a new lightweight `last_active_at` column on `profiles`.

### Consider: `last_active_at` on `profiles`

Cheap to add, useful for the Friends tab ("Active 2h ago"). Updated by any meaningful action (drill completion, arena battle, page-visit ping). One column:

```sql
ALTER TABLE public.profiles ADD COLUMN IF NOT EXISTS last_active_at TIMESTAMPTZ;
CREATE INDEX idx_profiles_last_active ON public.profiles(last_active_at DESC);
```

## Hooks API

### `useFriends()`

Single source of truth for the Friends tab.

```ts
type FriendStatus = "accepted" | "pending-incoming" | "pending-outgoing";

interface Friend {
  id: string;
  display_name: string;
  xp: number;
  elo: number;
  rank: ReturnType<typeof getRankFromElo>;
  streak: number;
  lastActiveAt: string | null;
  status: FriendStatus;
  acceptedAt: string | null;
}

function useFriends(): {
  friends: Friend[];             // status === "accepted"
  incomingRequests: Friend[];    // status === "pending-incoming"
  outgoingRequests: Friend[];    // status === "pending-outgoing"
  loading: boolean;
  sendRequest: (targetUserId: string) => Promise<void>;
  acceptRequest: (requesterId: string) => Promise<void>;
  declineRequest: (requesterId: string) => Promise<void>;
  removeFriend: (friendId: string) => Promise<void>;
  refresh: () => Promise<void>;
};
```

Internals:

1. Fetch `friendships` rows where `user_a = me` OR `user_b = me`
2. Extract the *other* user_id for each row
3. Single `profiles.select("id, display_name, xp, elo, last_active_at").in("id", otherIds)` query
4. Single `streaks.select("user_id, count").in("user_id", otherIds)` (works because of Option A above)
5. Single `user_xp.select("user_id, total_xp").in("user_id", otherIds)` if `user_xp` is the authoritative XP source (it is per `useMyXp`)
6. Stitch into `Friend[]` and partition by status

Cache via React Query (already in the app via `QueryClient`) keyed on `["friends", user.id]`.

### `useFriendSearch()`

```ts
function useFriendSearch(query: string): {
  results: { id: string; display_name: string; elo: number }[];
  loading: boolean;
};
```

Debounced 300ms. Queries `profiles.select("id, display_name, elo").ilike("display_name", `%${q}%`).neq("id", user.id).limit(10)`. Excludes the user and existing friends client-side.

### `useFriendInvite()`

```ts
function useFriendInvite(): {
  myInvites: { token: string; created_at: string; claimed_by: string | null }[];
  generateInvite: () => Promise<string>;   // returns the full shareable URL
  revokeInvite: (token: string) => Promise<void>;
};
```

`generateInvite` uses `crypto.getRandomValues(new Uint8Array(16))` → base64url → inserts the row → returns `${origin}/friends/invite/${token}`.

### Notification hook — reuse `arenaEmitter` pattern

```ts
// src/lib/events.ts — extend
type FriendsEvents = {
  "friends:request-received": { from: { id: string; display_name: string } };
  "friends:request-accepted": { by: { id: string; display_name: string } };

```

Wired up by a Supabase Realtime subscription in `useFriends` so the bell-icon chip in `SiteHeader` lights up the moment a new request arrives.

## Invitation flow (the tricky one)

### Signed-in invitee

1. User navigates to `/friends/invite/<token>`
2. `<InviteLandingPage>` fetches the invite via the standard authed RLS path: `friend_invites.select().eq("token", token).maybeSingle()` — `friend_invites_select_own` allows the inviter to read, but not random authed users. **Need a tweak:** add `OR true` to the SELECT policy for *unclaimed, non-expired* tokens specifically. Or simpler: gate validation behind a `claim_friend_invite(token text)` SECURITY DEFINER RPC.
3. The RPC: looks up the invite, checks expiry, inserts a `friendships` row (status `accepted` directly — the invite IS the consent), marks `claimed_by` + `claimed_at`. Returns the inviter's display_name for the success screen.
4. Page redirects to `/friends` with a toast: "You're now friends with Alice 🎉"

### Signed-out invitee

1. Public route renders. Without auth, calls a `peek_friend_invite(token text)` SECURITY DEFINER edge function that returns ONLY the inviter's display name (nothing sensitive). If invalid/expired → "Invite expired" screen.
2. "Sign up to accept" button → `/login?next=/friends/invite/<token>&signup=1`
3. `Login.tsx` already honors `location.state.from` for redirect — extend to also handle a query-param next URL.
4. After sign-up + email confirm flow, the user lands on `/friends/invite/<token>` again, now signed in → flow above kicks in.

The two pieces of new infrastructure here:

- **`claim_friend_invite(token text)`** — SECURITY DEFINER RPC, the only thing that can insert a friendship without going through the request/accept dance.
- **`peek_friend_invite(token text)`** — public RPC returning `{ inviter_display_name }` or null. Rate-limit it (existing supabase config or a simple in-function check on `pg_stat_statements`).

## Real-time

Subscribe to `friendships` changes scoped to the user via Supabase Realtime channels. Pattern is identical to `ArenaContext`'s `arenaChannel`:

```ts
useEffect(() => {
  if (!user) return;
  const channel = supabase
    .channel(`friends:${user.id}`)
    .on("postgres_changes", {
      event: "*",
      schema: "public",
      table: "friendships",
      filter: `or(user_a.eq.${user.id},user_b.eq.${user.id})`,
    }, () => refresh())
    .subscribe();
  return () => { supabase.removeChannel(channel); };
}, [user]);
```

Bell-icon notification:

- New row with `status='pending'` AND `requested_by !== me` → emit `friends:request-received`
- Existing row flips to `accepted` AND `requested_by === me` → emit `friends:request-accepted` (you-were-accepted toast)

Add a tiny `<FriendsBadge>` to `SiteHeader` next to the existing Arena notification, showing the count of incoming pending requests.

## Challenge integration (light hook into Arena)

The Arena context already has `sendDuelRequest(targetUserId, gamemode, prompt)`. The FriendCard's "Challenge" button calls `navigate(/arena?challenge=<friendId>&mode=standard)` and Arena reads the query params on mount to pre-select the friend + open the challenge modal.

Existing `acceptDuelRequest` flow handles the rest — no new infrastructure needed. The friendship just provides the discovery surface.

## Privacy & abuse

- **Search results** show only `display_name` + `elo` — not email, not last_active_at. Email is never exposed to other users.
- **`display_name` is user-controlled** — already true today. If we want to deter impersonation, add a uniqueness constraint later (post-MVP). For now, the ID disambiguates.
- **Removing a friend deletes both directions** (single row in the canonical ordering, so this is just one DELETE).
- **Pending request expiry**: an outgoing request older than 30 days could be auto-cleaned by a daily cron. Defer until we see real volume.
- **Block users**: cut from MVP, but the `status = 'blocked'` enum value is reserved in the schema so we don't need a migration to add it later.

## Risks & sharp edges

1. **`streaks` RLS change** breaks the existing model where streak is strictly private. Option A is correct but every other feature that touches `streaks` must keep working — audit `useSyncedStreak` and `useEventReminders` for assumptions.
2. **Realtime channel limits** — Supabase free tier caps concurrent channels. Reuse a single `friends:<uid>` channel; don't open one per friend.
3. **Invite token URL is sensitive** — anyone with the URL becomes a friend. 14-day expiry mitigates. Don't email it server-side (defer email-invite to v2). Inviter can revoke any time.
4. **Display name uniqueness** — search by `display_name` could surface "Alex" duplicates. Show ELO/rank under each result so the user picks the right one. Long-term: add a `username` column (unique, lowercase, /^[a-z0-9_]{3,20}$/).
5. **Migration ordering** — `friendships` must exist before the streak policy extension references it. Single migration file with both, or two files numbered in order.
6. **`crypto.getRandomValues` availability** — fine in all evergreen browsers and Node 18+, but worth a `typeof crypto.getRandomValues === "function"` guard at the call site.
